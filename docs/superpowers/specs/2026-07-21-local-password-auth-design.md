# 本地密码认证设计

日期：2026-07-21

## 1. 背景与目标

Pi Agent Web 当前没有应用级访问认证。打开服务后，能够连接到该地址的人可以访问页面以及会话、文件、模型配置和 Agent 等接口。现有 `app/api/auth/*` 只负责模型提供商的 OAuth 与 API Key 管理，不是 Pi Agent Web 自身的登录功能。

本次增加一个适合单人、本地或自托管环境使用的简单密码门禁：

- 不需要用户名，只输入一个本地配置的密码。
- 默认保护整个网页和所有业务 API。
- 密码主要通过 Pi Agent 数据目录中的专用配置文件管理。
- 环境变量可以覆盖配置文件，方便 Docker、服务器和临时启动。
- 未设置密码时保持锁定，避免服务意外无保护开放。
- 用户可以通过明确配置 `disabled: true` 绕过认证。

本次不做多用户、账号管理、找回密码、验证码、第三方登录或网页修改密码。

## 2. 配置模型

### 2.1 配置文件位置

默认配置文件：

```text
~/.pi/agent/pi-web.json
```

如果设置了 `PI_CODING_AGENT_DIR`，配置文件跟随该目录：

```text
$PI_CODING_AGENT_DIR/pi-web.json
```

示例：

```json
{
  "auth": {
    "password": "replace-with-a-strong-password",
    "disabled": false
  }
}
```

建议将文件限制为当前用户可读：

```bash
chmod 600 ~/.pi/agent/pi-web.json
```

### 2.2 环境变量覆盖

支持：

```bash
PI_WEB_PASSWORD=replace-with-a-strong-password
PI_WEB_AUTH_DISABLED=true
```

每个环境变量只覆盖自己的字段，优先级为：

1. 环境变量
2. `pi-web.json`
3. 安全默认值

`PI_WEB_AUTH_DISABLED` 只接受明确的 `true` 或 `false`。非法值视为配置错误，不会放行。

### 2.3 最终状态

配置读取后只能得到以下三种状态：

1. **已禁用认证**：`disabled === true`，所有请求直接放行。
2. **认证已启用**：`disabled !== true` 且存在非空密码，需要登录。
3. **配置未就绪**：未禁用认证，但没有有效密码，应用保持锁定并显示配置说明。

配置文件不存在等同于“配置未就绪”。配置文件损坏、字段类型错误或环境变量值非法属于“配置错误”，同样保持锁定。

## 3. 整体架构

### 3.1 模块划分

认证功能拆为几个边界清楚的部分：

- `lib/web-auth-config.ts`
  - 确定 Pi Agent 数据目录。
  - 读取并校验 `pi-web.json`。
  - 合并环境变量覆盖。
  - 返回不含歧义的认证状态与可公开的错误类型。

- `lib/web-auth-session.ts`
  - 比较登录密码。
  - 创建、解析和校验登录凭证。
  - 统一 Cookie 名称、有效期和属性。
  - 不向客户端暴露配置密码。

- `proxy.ts`
  - 在页面或接口进入实际处理逻辑前检查认证状态。
  - 区分网页跳转和 API 的 `401` 响应。
  - 放行登录相关路径及必要静态资源。

- `app/login/page.tsx` 与登录表单组件
  - 根据公开状态显示密码输入框、配置提示或配置错误。
  - 登录成功后跳回用户原本想访问的站内地址。

- `app/api/gate/login/route.ts`
  - 接收密码，完成比较，成功时设置 Cookie。
  - 记录短时间内的连续失败并加入等待。

- `app/api/gate/logout/route.ts`
  - 删除 Cookie。

- `app/api/gate/status/route.ts`
  - 只返回登录页和主界面需要的公开状态，例如 `enabled`、`disabled`、`unconfigured` 或 `error`。
  - 不返回密码、密码摘要或内部错误细节。

- 主界面退出入口与认证关闭提示
  - 主界面提供退出按钮。
  - 明确绕过认证时显示醒目提示和实际配置文件路径。

### 3.2 路径命名

现有 `/api/auth/*` 保持原用途，不作改名或复用。应用门禁使用：

```text
/login
/api/gate/status
/api/gate/login
/api/gate/logout
```

这样可避免把“登录 Pi Agent Web”和“登录模型提供商”混淆。

## 4. 请求流程

### 4.1 访问网页

1. `proxy.ts` 读取有效认证配置。
2. 如果认证被明确禁用，请求继续，并把禁用状态传给应用用于显示警告。
3. 如果认证已启用且 Cookie 有效，请求继续。
4. 如果认证已启用但 Cookie 无效，跳转到：

```text
/login?next=<原站内路径>
```

5. 如果配置未就绪或配置错误，跳转到 `/login`，由登录页显示配置说明。
6. 已登录用户访问 `/login` 时跳回安全的 `next` 地址；没有有效 `next` 时回到 `/`。

### 4.2 访问 API

- `/api/gate/status`、`/api/gate/login` 和 `/api/gate/logout` 可在未登录时访问。
- 其他 API 在认证开启但未登录时返回：

```json
{
  "error": "Unauthorized"
}
```

HTTP 状态码为 `401`，不会重定向到 HTML 登录页。

- 配置未就绪或配置错误时，业务 API 返回 `503`，使用不泄露敏感信息的错误码，例如 `AUTH_NOT_CONFIGURED` 或 `AUTH_CONFIG_ERROR`。
- 实时事件连接、会话管理、文件读取、模型配置、技能、插件、工作树等现有业务接口全部受保护。

### 4.3 登录

1. 浏览器向 `/api/gate/login` 提交 JSON：

```json
{
  "password": "...",
  "next": "/original/path"
}
```

2. 服务端重新读取当前有效配置。
3. 如果认证被禁用，返回安全的站内跳转地址，不创建 Cookie。
4. 如果配置未就绪或有错误，拒绝登录并返回公开状态。
5. 服务端使用固定时间方式比较提交密码与配置密码。
6. 密码错误时返回统一的“密码不正确”，不说明密码长度或其他细节。
7. 密码正确时创建有签名和到期时间的 Cookie，并返回安全跳转地址。
8. 客户端使用返回的站内地址完成跳转。

### 4.4 退出

1. 主界面退出按钮向 `/api/gate/logout` 发送 `POST`。
2. 服务端通过 `Max-Age=0` 删除认证 Cookie。
3. 客户端进入 `/login`。

## 5. 登录凭证与 Cookie

### 5.1 凭证内容

Cookie 不保存明文密码。凭证包含：

- 格式版本
- 到期时间
- 随机值
- 对上述内容生成的 HMAC-SHA256 签名

签名密钥通过当前密码和固定用途标签派生。这样：

- 服务重启后登录状态仍可保持。
- 修改密码后，旧 Cookie 的签名立即失效。
- 客户端不能自行延长有效期或修改内容。

密码比较、签名比较均使用固定时间比较，并正确处理长度不同的输入，避免抛出异常。

### 5.2 Cookie 属性

Cookie 名称使用专用常量，例如 `pi_web_session`。默认设置：

- `HttpOnly: true`
- `SameSite: Lax`
- `Path: /`
- `Max-Age: 30 days`
- HTTPS 请求时设置 `Secure: true`

本项目经常直接运行在本地 HTTP 地址，因此不能无条件设置 `Secure`，否则本地登录无法工作。部署到非本机网络时，文档应建议在 HTTPS 反向代理后使用。

## 6. 登录失败限制

这是本地单人门禁，不引入数据库。登录接口在当前服务进程中按客户端地址保存短期失败记录：

- 连续失败后逐步增加短暂等待。
- 等待时间设置上限，避免无限阻塞。
- 登录成功后清除该地址的失败记录。
- 旧记录自动过期并清理。
- 客户端只收到统一错误，不显示当前失败次数。

该限制用于降低快速批量尝试，不能替代强密码、HTTPS 和网络访问控制。服务重启会清空失败记录，这是本地轻量方案可接受的限制。

## 7. 登录页面与主界面

### 7.1 登录页

登录页沿用项目现有颜色变量和字体，保持简单：

- 标题：`Pi Agent Web`
- 认证正常时显示一个密码框和登录按钮。
- 密码错误时显示统一错误。
- 提交过程中禁用按钮，防止重复提交。
- 配置未就绪时不显示无效登录框，而是显示配置文件实际路径、示例 JSON、环境变量示例和重启提示。
- 配置错误时显示“认证配置无法读取”，页面只提供安全的修复说明，具体错误写入服务端日志。

### 7.2 主界面

- 在现有顶部栏加入退出入口。
- 只有认证已启用时显示退出按钮。
- 认证被明确禁用时显示醒目的“认证已关闭”提示，并显示配置文件路径，方便用户意识到当前服务没有门禁。
- 提示在桌面端显示文字，在空间较小的移动端使用图标和可读的辅助标签。

## 8. 安全边界与异常处理

### 8.1 安全跳转

`next` 只接受以单个 `/` 开头的站内路径，拒绝：

- `//example.com`
- 完整的 `http://` 或 `https://` 地址
- 无法解析或包含异常控制字符的值
- `/login` 自身造成的循环跳转

不合法时统一回到 `/`。

### 8.2 放行路径

`proxy.ts` 只放行：

- `/login`
- `/api/gate/status`
- `/api/gate/login`
- `/api/gate/logout`
- Next.js 必需的静态资源和公开资源

静态资源通过固定 `matcher` 排除，避免登录页缺少脚本、样式或字体。所有业务 API 保持在认证范围内。

### 8.3 配置错误

- 文件不存在：未配置状态。
- JSON 无法解析：配置错误。
- `auth`、`password` 或 `disabled` 类型不正确：配置错误。
- 密码是空字符串或只有空白：未配置状态。
- 文件读取权限错误：配置错误。
- `PI_WEB_AUTH_DISABLED` 非 `true`/`false`：配置错误。

这些情况都不会自动放行业务页面和 API。服务端日志可以记录文件路径和具体原因，但浏览器只显示可安全公开的信息。

### 8.4 绕过模式

`disabled: true` 或 `PI_WEB_AUTH_DISABLED=true` 是唯一的无密码放行方式。它必须是用户明确配置的值；缺少配置不能等同于绕过。

绕过模式只关闭 Pi Agent Web 新增的应用门禁，不改变现有模型提供商 OAuth、API Key 或文件访问规则。

## 10. 测试与验收

### 10.1 配置测试

- 默认目录和 `PI_CODING_AGENT_DIR` 自定义目录正确。
- 文件配置可以被读取。
- 两个环境变量分别覆盖对应字段。
- 未配置密码时保持锁定。
- `disabled: true` 明确放行。
- 无效 JSON、错误字段类型、无效布尔值和读取失败保持锁定。

### 10.2 凭证测试

- 正确密码登录成功，错误密码失败。
- Cookie 不包含明文密码。
- 有效 Cookie 可以通过校验。
- Cookie 被修改、签名错误或已过期时校验失败。
- 修改密码后旧 Cookie 失败。
- 不同长度的错误输入不会让固定时间比较抛出异常。

### 10.3 请求保护测试

- 未登录访问页面会跳转 `/login` 并保留安全的 `next`。
- 未登录访问业务 API 返回 `401`。
- 未配置或配置错误时页面进入配置说明，业务 API 返回 `503`。
- 登录页、三个 gate 接口和必要静态资源不会产生循环跳转。
- 已登录访问 `/login` 回到首页或安全 `next`。
- 非法外部 `next` 被替换为 `/`。
- 认证关闭时页面和 API 均放行。
- 现有 `/api/auth/*` 模型提供商接口也受应用门禁保护。

### 10.4 界面测试

- 登录成功回到原页面。
- 错误密码显示统一提示。
- 退出后再次访问业务页面需要登录。
- 认证关闭时主界面有清晰警告。
- 桌面和移动尺寸都可以操作登录与退出。

### 10.5 项目检查

实施完成后运行：

```bash
node --test <新增认证测试文件>
node_modules/.bin/tsc --noEmit
npm run lint
```

开发阶段不运行 `next build`，避免污染 `.next/` 并影响 `npm run dev`。

## 11. 文档要求

README 中应说明：

- 配置文件的默认和自定义位置。
- 密码配置示例。
- 环境变量覆盖方式。
- 如何明确关闭认证。
- 修改配置后需要重启服务才能确保所有进程使用同一配置。
- 局域网或公网使用时应采用强密码、HTTPS 和防火墙/反向代理限制。
- `disabled: true` 会让整个 Pi Agent Web 恢复无门禁访问。

## 12. 完成标准

满足以下条件即视为本功能完成：

- 默认未配置时应用不会无保护开放。
- 配置一个密码后，网页和业务 API 必须登录才能访问。
- 正确密码可登录 30 天，修改密码使旧登录状态失效。
- 可以通过明确配置关闭认证，并在界面看到警告。
- 登录、退出、原路径返回和接口错误行为一致且可测试。
- 不破坏现有模型认证、会话、文件、SSE、技能、插件和工作树功能。
- README 提供足够的本地和部署配置说明。
